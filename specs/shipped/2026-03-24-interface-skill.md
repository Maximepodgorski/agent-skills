# Interface Skill

## Context

Build a sophisticated skill that generates SaaS/Mobile UI screens by composing existing components from a codebase. The spec action proposes multiple ASCII wireframe layout options in chat, lets the user iterate until satisfied, then specs the chosen layout with workflow-level rigor.

Core guarantee: **Zero component reinvented** — if an existing component covers the use case, reuse it. Gaps are explicitly decided by the user.

### 3-Layer Knowledge Model


### Spec Action — 2-Phase Design

The spec action is the core of the skill. It combines layout proposal (compose) and specification in a single action with 2 phases.

**Phase 1: COMPOSE (iterative, max 3 rounds)**

```
/interface spec "settings page"
     │
     ├─ Load conventions + live scan components
     ├─ Load principles: layout-patterns + composition + responsive
     │
     ▼
  Round 1: Propose 3 ASCII wireframe options
     │     (each with coverage %, component mapping, missing flags)
     │
     ├─ User choisit ──────────────────────────────────▶ PHASE 2
     │
     ├─ User demande variante ("plus compact", "avec tabs", "mix 1+3")
     │
     ▼
  Round 2: Nouvelles options (affinées par feedback)
     │
     ├─ User choisit ──────────────────────────────────▶ PHASE 2
     │
     ├─ Toujours pas satisfait
     │
     ▼
  Round 3: Dernière tentative
     │
     ├─ User choisit ──────────────────────────────────▶ PHASE 2
     │
     └─ Toujours non ─▶ "Describe the exact layout" ─▶ PHASE 2
                         (free-form fallback)
```

Between rounds, the agent refines proposals based on user feedback without re-scanning. Component inventory stays in memory.

**Phase 2: SPEC (one-shot from compose-block)**

The chosen layout is captured as an internal compose-block. Phase 2 generates the full spec from it — no re-analysis, no re-scan.

**Skip-if conditions (keep spec.md manageable):**

```
PHASE 1:
  SKIP Figma extraction     IF no Figma URL provided
  SKIP round 2-3            IF user picks in round 1
  SKIP missing components    IF coverage = 100%
  SKIP free-form fallback   IF user picks before round 3

PHASE 2:
  SKIP State Machine section IF screen is stateless (simple form, static page)
  SKIP Data Flow section     IF screen has no dynamic data (static content page)
  SKIP Test Strategy section IF tier = mini
  SKIP Risks section         IF tier = mini
  SKIP Analysis section      IF tier = mini

TIER DETECTION (auto):
  coverage >= 90% AND components <= 8 AND no custom data flow → MINI
  everything else → STANDARD
```

### Spec-Review — Dynamic Perspective Selection

Follows the same pattern as workflow spec-review: dynamic perspective selection based on spec content, parallel sub-agents, action-driven findings, optional merge.

**Process:**
1. Load spec from `ds/interfaces/active/`
2. Read spec's context, component coverage, layout architecture, screen type
3. Auto-select 3-6 perspectives relevant to THIS screen's domain
4. Always include 1 **Skeptic** (domain-agnostic adversarial thinker)
5. Present selected perspectives to user before launching
6. Launch ALL as parallel sub-agents (single message)
7. Synthesize findings → verdict
8. Optional: merge findings into spec

**Perspective selection by screen type:**

| Screen Type | Perspectives |
|-------------|-------------|
| Dashboard | Data Visualization Expert, UX Designer (information density), Performance Engineer, Skeptic |
| Form / Settings | Form UX Expert (validation, progressive disclosure), DS Consistency Reviewer, Frontend Engineer, Skeptic |
| List / Table | Data UX Expert (pagination, filtering, empty states), Performance Engineer, Accessibility Specialist, Skeptic |
| Mobile screen | Mobile Specialist (touch targets, gestures, safe areas), UX Designer (thumb zones), Frontend Engineer, Skeptic |
| Landing / Marketing | UX Designer (visual hierarchy, CTA), Performance Engineer (LCP, above-the-fold), Conversion Specialist, Skeptic |
| Auth / Onboarding | UX Designer (friction, drop-off), Security Engineer (input validation), Accessibility Specialist, Skeptic |

**Per-agent prompt (4 questions — same as workflow):**

```
You are a {Role}. Review this interface spec adversarially.

SPEC: {full spec content}
CONTEXT: {component files from coverage table, conventions.md}

Answer these 4 questions:

1. ASSUMPTIONS: What's assumed that could be wrong?
   | Assumption | Evidence For | Evidence Against | Verdict |
   (min 2 per perspective)

2. BLIND SPOTS: What hasn't been considered?
   - 2-3 items with [category] + why it matters

3. FAILURE HYPOTHESES:
   | IF | THEN | BECAUSE | Severity |
   (1-3 per perspective)

4. THE REAL QUESTION: Is this solving the right problem from your perspective?

Every finding must end with: → update spec / → explore / → question / → no action
```

**Verdict algorithm:**
- ALL perspectives positive, no HIGH severity → **APPROVED**
- Any CRITICAL finding → **BLOCKED** (resolve before build)
- No CRITICAL but >3 HIGH → **NEEDS REVISION** (fix spec, re-review max 2 rounds)
- Otherwise → **APPROVED WITH NOTES**

### Figma Integration (Optional Input)

spec accepts a Figma URL as optional input for its compose phase.

```
/interface spec "settings page"                              ← text only
/interface spec figma.com/design/abc/file?node-id=1:2        ← Figma frame
/interface spec "settings page" figma.com/design/...         ← both
```

When Figma URL provided:
1. Parse URL → extract fileKey + nodeId
2. Call `get_design_context(fileKey, nodeId)` → code + screenshot + hints
3. Extract layout structure from Figma frame
4. Map Figma children to component inventory → precise coverage %
5. Fallback: if Figma MCP unavailable → continue with text description only

## User Journey

### Primary Journey

ACTOR: Developer building a new screen
GOAL: Compose a UI screen from existing components without starting from scratch
PRECONDITION: Project has components in codebase. init has run (or auto-triggers).

1. User runs `/interface spec "settings page"`
   → System checks for ds/conventions.md (if missing → auto-triggers init)
   → System loads conventions + runs live scan for components
   → **Phase 1 (compose):** User sees 3 ASCII wireframe options with coverage % and component mapping

2. User picks option (e.g., "option 2" or "mix 1+3" or "more compact")
   → If satisfied → system captures compose-block, proceeds to Phase 2
   → If not → system proposes refined options (max 3 rounds, then free-form fallback)

3. **Phase 2 (spec):** System generates full spec from compose-block
   → No re-scan, no re-analysis — enriches the compose-block with journey, ACs, states, data flow
   → User sees spec file saved at ds/interfaces/active/ + chat summary

4. User runs `/interface spec-review`
   → System reads spec, detects screen type, selects 3-6 perspectives
   → System presents perspectives to user for confirmation
   → System launches all perspectives as parallel sub-agents
   → User sees synthesized report with verdict (APPROVED / NEEDS REVISION / BLOCKED)

5. User runs `/interface build`
   → System enters ship loop, iterating on spec items
   → System checks off spec items as implemented
   → User sees progress updates per iteration

6. User runs `/interface review`
   → System checks reuse compliance, consistency, a11y
   → User sees verdict with CRITICAL/WARN findings

POSTCONDITION: Screen implemented, spec items checked, review passed.

### Figma Journey (Variant)

ACTOR: Developer with a Figma design to implement
PRECONDITION: Figma frame URL available.

1. User runs `/interface spec figma.com/design/abc/...`
   → System calls get_design_context → extracts layout + components
   → System maps Figma children to existing DS components
   → Phase 1: User sees 2-3 options (faithful reproduction vs optimized compositions) with precise coverage %
   → User picks → Phase 2 generates spec

### Error Journeys

E1. No conventions.md found
    Trigger: First-time use, init never ran
    1. User runs `/interface spec "dashboard"`
       → System detects missing conventions.md
       → System auto-triggers init (parallel scan)
       → User sees "First run detected. Scanning project..." then spec continues
    Recovery: Seamless — init runs, then spec proceeds with Phase 1

E2. Zero components found in codebase
    Trigger: New project with no DS yet
    1. User runs `/interface spec "login page"`
       → System scans, finds 0 existing components
       → User sees "No components found. This skill works best with an existing component library. Propose component list for this screen?"
    Recovery: Skill proposes which components to create first (delegates to /component skill)

E3. Figma MCP unavailable
    Trigger: Figma URL provided but MCP not configured
    1. User runs `/interface spec figma.com/design/...`
       → System attempts get_design_context, fails
       → User sees "Figma MCP not available. Continuing with text description."
    Recovery: Falls back to text-only compose phase

E4. User rejects all compose options (3 rounds exhausted)
    Trigger: None of the proposed layouts match the user's vision
    1. System exhausts 3 rounds of proposals
       → User sees "Describe the exact layout you want (component list + arrangement)"
       → User provides free-form description
       → System builds compose-block from free-form input
    Recovery: Free-form fallback — never blocks

### Edge Cases

EC1. All components exist (100% coverage): Skip "missing components" section in spec
EC2. User provides both Figma + text description: Figma takes priority, text refines intent
EC3. Spec already exists for this screen: Ask user — overwrite / update / cancel
EC4. conventions.md is stale (>30 days): Warn user, suggest re-init
EC5. User says "option 2 but with X changed": Agent produces a variant of option 2, counts as same round

## Analysis

### Assumptions Challenged

| Assumption | Evidence For | Evidence Against | Verdict |
|------------|-------------|------------------|---------|
| ASCII wireframes are sufficient to convey layout intent | Terminal-native, instant, no tooling needed | Complex responsive layouts hard to show in ASCII | RISKY — mitigate with component tree + Figma screenshot |
| 1 persisted file (conventions.md) is enough | Conventions rarely change, live scan covers the rest | Patterns section might get stale, agent loses layout vocabulary between sessions | VALID — patterns are in skill's principles/, not in conventions.md |
| 3 compose rounds is enough before free-form | Most users pick in round 1-2; round 3 = edge case | User might want unlimited iteration | VALID — free-form fallback prevents blocking |
| Live scan is fast enough for compose phase | Grep/glob are fast on modern SSDs; scan happens once, stays in memory across rounds | 200+ component codebases could be slow on first scan | RISKY — mitigate with category filtering |
| Merging compose into spec simplifies the flow | 1 trigger instead of 2; no handoff friction; daily use = spec → build | Some users might want to explore layouts without committing to spec | VALID — compose phase is non-committal; spec file only generated after user picks |

### Blind Spots

1. **[Integration]** How does interface skill interact with component skill when a missing component is flagged? Does it pause build? Create a stub? The handoff to /component is mentioned but not protocoled.
   Why it matters: Without clear handoff, user gets stuck between two skills.

2. **[Scale]** No guidance on multi-section screens (e.g., dashboard with 4 independent zones). Does compose propose the full screen or zone by zone?
   Why it matters: Dashboard-type screens are the most common SaaS use case.

3. **[Persistence]** If the conversation is interrupted during Phase 1 (compose loop), the component inventory and round state are lost. Phase 2 output (spec file) is persisted, but Phase 1 is ephemeral.
   Why it matters: Long compose iterations on complex screens could be lost.

### Failure Hypotheses

| IF | THEN | BECAUSE | Severity | Mitigation |
|----|------|---------|----------|------------|
| Component APIs change after conventions.md was generated | compose phase proposes layouts with wrong props/slots | conventions.md doesn't track component APIs (live scan does) | LOW | Live scan reads actual files, not conventions.md |
| Figma frame has deeply nested auto-layout | get_design_context returns flat component list without layout hierarchy | Figma MCP returns code, not layout tree | MED | Use screenshot + code hints to reconstruct layout; fallback to text |
| User's project uses a non-standard component structure (no clear atoms/molecules) | init fails to classify components | Classification logic assumes Atomic Design | HIGH | init should detect and adapt to project's own hierarchy, not impose Atomic Design |
| Compose phase generates 3 nearly identical options for simple screens | User feels options are fake/useless | Simple screens have limited layout variation | MED | For simple screens (high coverage, few components), propose 2 options instead of 3; include 1 "minimal" variant |

### The Real Question

Not "how do we generate UI screens?" but "how do we make the agent see the project's component library the same way a senior developer does — knowing what exists, what fits, and what's missing?"
